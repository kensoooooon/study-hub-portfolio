# Django Study Hub

> **Note:** このリポジトリは本番プロダクトのサニタイズ版です。クライアント情報・本番認証情報を除去しています。実際の開発履歴（Issue・コミット）はプライベートリポジトリで管理されており、ここでのコミットは機能単位の同期スナップショットです。

## このプロジェクトについて

Django Study Hub は、LINE Messaging API と OpenAI API を組み合わせた教育向け SaaS プラットフォームです。英単語・リスニング・リーディング・数学などの学習コンテンツを提供し、SuperMemo-2 アルゴリズムによるスペースドリピティションと、LINE を通じた AI 質問応答・リマインダー配信を実装しています。複数の教育機関に実際に展開し、継続稼働中です。

---

## なぜ作ったか

きっかけは、学習塾での課題でした。生徒からの質問に答えるために LINE 公式チャンネルを作ったものの、うまく機能しませんでした。

問題の構造はシンプルです。専属の担当者がいない、1問答えるごとの報酬はあるが大きなインセンティブにはならない、深夜に質問が来ても誰も気づかない——。中小の規模では人手や強制で解決できる問題ではありませんでした。

解決策として「人に頼らない仕組み」を選びました。それが OpenAI API との連携です。

---

## 汎用 AI との差別化

AI 連携で Q&A を実装したあと、「これは ChatGPT を使えばよくない？」という問いに直面しました。2024 年時点でモデルの性能はすでに十分高く、汎用的な機能はコモディティ化が進む一方でした。

差別化の答えは、**大企業が絶対にやらない・やれないニッチな部分に集中すること**です。

- 教科書完全準拠の英単語データベース（市販アプリでは細かすぎる粒度）
- 生徒の質問履歴を講師と共有する機能
- 学習履歴の蓄積と教室単位での管理

独自のコンテンツとデータの蓄積が、汎用 AI との違いを生む部分だと考えています。

## 主要な設計判断

### LINE を通知・対話チャネルとして採用した理由

**課題：** 教育用アプリを別途インストールしてもらう前提では、学生の参加率が下がる。

**選択：** 日本国内の普及率がきわめて高い LINE を活用することで、学習アプリ導入の障壁をなくす。

**トレードオフ：** UI の自由度は Web アプリに劣る。ただし、生徒が日常的に使うツールに入り込む方が継続利用の観点では優位と判断しました。現に、公式 LINE への質問は毎日継続して届いており、元々の課題だった深夜の未回答問題は解消されています。

---

### GCP（Google App Engine）を選んだ理由

**課題：** 個人開発・小規模運用での初期コスト抑制と、複数のマネージドサービスを組み合わせた構成の必要性。

**選択：** App Engine の無料枠活用と、Cloud KMS・Secret Manager・Pub/Sub とのシームレスな統合。

**トレードオフ：** ベンダーロックインのリスクはある。しかし、独自インフラを構築するより、マネージドサービスを使ってプロダクト開発に集中する方が合理的と判断しました。

---

### SuperMemo-2 を採用した理由

**課題：** 単純な「問題を解く」体験では継続的な記憶定着につながらない。

**選択：** 忘却曲線に基づくスペースドリピティション（SuperMemo-2）を、全トレーナーモジュールに共通実装。`ease_factor`（記憶の容易さ係数）・`interval`（次回復習までの間隔）・`next_due_at`（次回復習予定日時）を管理。

**トレードオフ：** 実装が複雑になる。しかし、「やりっぱなし」の学習ではなく「必要なタイミングで復習する」仕組みを提供することが、教育的に重要と判断しました。

---

### 非同期リマインダー配信（Cloud Pub/Sub）

**課題：** リマインダー配信を Web リクエスト内で同期処理すると、タイムアウトや遅延が発生しうる。

**選択：** Cloud Pub/Sub を用いた非同期アーキテクチャ。`trigger_reminders.py` がキューにメッセージを送り、独立した `listener.py` が受信・LINE 配信処理を担う。

**トレードオフ：** 構成が複雑になる。ただし、Web プロセスとリマインダー配信を分離することで、互いの障害が波及しない設計になります。

---

### LINE チャネル認証情報のエンベロープ暗号化

**課題：** マルチテナント構成では、組織ごとに異なる LINE チャネルの秘密鍵を共有 DB 上で安全に保管する必要がある。

**選択：** Cloud KMS を用いたエンベロープ暗号化（DEK で平文を暗号化し、KMS 管理の KEK で DEK を暗号化）。

**トレードオフ：** KMS へのラウンドトリップが加わり、実装も複雑になる。秘密鍵を平文で共有 DB に保管するリスクを避けるためには、この層の分離が必要でした。

## アーキテクチャ概要

```
利用者（学生 / 教師 / 管理者）
    │ LINE アプリ / Web ブラウザ
    ▼
Google App Engine
    │ Django 5.1.3
    ├── accounts（RBAC・マルチテナント管理）
    ├── line_channels（エンベロープ暗号化）
    ├── processors（OpenAI API 連携）
    ├── vocab / listening / read / math_trainer（学習モジュール）
    └── study_reminder（リマインダー管理）
    │
    ├──→ Cloud SQL（PostgreSQL）
    ├──→ Cloud Pub/Sub ──→ listener.py ──→ LINE API
    └──→ OpenAI API / Cloud KMS / Secret Manager
```

セキュリティ基盤：Cloud KMS（エンベロープ暗号化）/ Secret Manager（機密情報管理）/ HSTS・CSRF・XSS 対策

## 開発プロセス

Issue 駆動開発を採用しています。1 つの Issue が 1 つの変更単位となり、スコープの拡大を防ぎます。

```
Issue 作成（自分の言葉で）
  → ChatGPT と壁打ち（背景・選択肢の整理）
  → 仕様確定
  → Claude に調査と実装プランを依頼
  → ChatGPT でプランレビュー（セカンドオピニオン）
  → Claude に実装を依頼
  → コードレビュー → テスト → マージ
```

**AI との役割分担：** ChatGPT は「壁打ち・レビュー」、Claude は「調査・実装」として使い分けています。「なぜ作るか」「どう設計するか」は自分が決め、「何を実装するか」を AI に渡す分業が基本です。

最近は、「枠組みと方向性を先に自分で提案してから AI に渡す」方向に進化しています。AI への依存度を下げながら、設計判断の主体を自分に置くための意識的な調整です。

## 運用経験と改善

### うまくいっていること

- **毎日の Q&A 継続：** LINE への質問が毎日継続して届いており、深夜帯の未回答問題は解消されました
- **リマインダー配信：** Cloud Pub/Sub による非同期配信は安定稼働しています
- **システムの継続稼働：** 大規模なインフラ運用経験がない状態から本番環境を立ち上げ、継続稼働させること自体が一つの成果です

### 現在の課題

**コンテンツへのアクセス誘導**

LINE に流入するユーザーを、トレーナーなどの独自コンテンツに誘導することがまだできていません。最近はリマインダーにリンクを付けることを試みましたが、効果の計測環境が整っていないため、改善できているかどうかが不明な状態です。

**計測インフラの不在**

どの機能がどれだけ使われているかを可視化できていないため、開発優先順位の判断が直感頼みになっています。これは現在最も急いで改善すべき点です。

---

## 今やり直すなら

**1. Organization を最初から必須フィールドにする**

開発初期に Organization をオプション扱いにしていたため、後に必須化しようとした際に問題が発生しました。既存の所属組織なしユーザーへの救済経路が必要となり、その救済ロジックがマルチテナントのデータ境界を曖昧にしています。現在もその影響でマイグレーション周りの慎重な作業が続いています。

スキーマ設計は後から変えるコストが大きい。マルチテナント構成では特に、テナント境界の設計は最初が肝心です。

**2. 利用可視化を初期から実装する**

開発初期は「自分が必要だと思うものを追加する」フェーズがありました。結果として、「次に何を作るべきか」の根拠になるデータがない状態です。機能より先に計測の仕組みを作るべきでした。

## 技術スタック

| 分類 | 技術 | 選定理由 |
|------|------|----------|
| バックエンド | Python / Django 5.1.3 | 習熟度・エコシステム |
